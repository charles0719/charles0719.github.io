分享一个有意思的项目 any-agent：一组帮助更好使用 Claude Code 的 Skill 集合

发表于 2026-04-05 更新于 2026-04-06 分类于 技术分享 本文字数： 3.9k 阅读时长 ≈ 4 分钟

最近在 GitHub 上看到一个项目：any-agent。

因为我自己本身就对 Claude Code、Agent、Skill 这些方向比较感兴趣，所以看到这种项目时，通常都会顺手往下看一看。

看完之后我觉得，any-agent 挺适合单独拿出来聊一篇。它不是那种传统意义上“启动起来就能看到页面”的应用，而是一组围绕 Claude Code 使用体验做增强的 Skill 集合。

更具体一点说，我会把它理解成：

一个帮助你更好使用 Claude Code 的项目。

它的主要价值不是替你直接执行业务，而是把一些和 Agent SDK、session、custom tools、插件市场相关的知识和方法组织成更容易命中的 Skill，让 Claude 在这些主题下答得更准、更稳。

所以这篇文章我想重点讲清楚：

  • any-agent 主要提供了哪些能力
  • 这些 Skill 对使用者到底有什么帮助
  • 装到本地之后，平时应该怎么用
  • 为什么它对经常研究 Claude Code 的人有价值

先说结论：Skill 不是“新功能按钮”，而是“更容易命中正确能力”

这是我研究下来最重要的体感。

装了 any-agent 之后，你并不会得到一个新的 Web 页面，也不会突然出现一个“帮你查 Redis”的独立程序。

你真正得到的是：

  • Claude Code 在某些主题下更容易走对方向
  • 同样一个问题，更容易命中正确的方法
  • 对复杂主题的回答更稳定，不容易漏步骤

所以 Skill 的价值，不是“凭空创造能力”，而是：

帮助 Claude 更稳定地使用已有能力。

没有 Skill 和有 Skill，区别到底在哪

这个对比如果不讲清楚，读者很容易觉得 Skill 很虚。

我现在的理解是这样的。

没有 Skill 时

你当然也可以直接问 Claude：

  • “怎么给 Agent SDK 加 custom tools？”
  • “怎么恢复 session？”
  • “怎么接 MCP server？”

Claude 也有可能答出来。

但问题是：

  • 它有时会泛泛而谈
  • 有时会漏步骤
  • 有时会把几个层级混在一起
  • 有时会不知道你现在到底是在问 tools、session 还是 advanced 配置

换句话说，它能答，但不一定每次都答得稳。

有 Skill 时

当你装了 any-agent，Claude 更容易：

  • 把你的问题归到正确主题
  • 使用更贴近这个主题的讲解方式
  • 更完整地覆盖关键步骤
  • 在复杂问题里少走弯路

所以 Skill 最有价值的地方，不是“替你干活”，而是：

让 Claude 更容易正确理解你现在到底在问什么。

装完之后，实际是怎么用的

这也是最关键的使用者视角问题。

装完 any-agent 之后，你日常使用上主要有两种方式：

1. 自然语言直接问

这是最常见也最适合大多数人的方式。

比如你可以直接问：

  • “怎么给 Claude Agent SDK 加自定义工具？”
  • “怎么恢复 Claude Agent SDK 的 session？”
  • “怎么做 structured output？”
  • “怎么搭一个 Claude Code 插件市场？”

这时候，如果你的提问命中了某个 Skill 的描述，Claude 就更容易自动走到那个 Skill 对应的知识路径。

2. 手动显式调用某个 Skill

如果你已经知道自己要哪一个 Skill，也可以更明确地叫它出来。

例如这类形式：

这个方式更适合你已经很明确自己要什么的时候。

所以从使用者角度来说，Skill 的调用方式并不复杂，本质上就两类：

  • 自然语言问，等它自动命中
  • 你自己明确指定 Skill

以 custom tools 为例，Skill 到底帮了什么

这是我自己最有感触的一个点。

假设你现在想做的是：

让 Claude Agent SDK 接一个 tool，去查询 Redis 里的业务状态。

如果没有 Skill，你可能会这样问：

Claude 也许会答，但有时候会出现这些问题：

  • 开始泛泛解释 Redis 是什么
  • 把 MCP server 和 tool 混在一起讲
  • 没有讲清楚 tool()、createSdkMcpServer()、mcpServers 之间的顺序

但有了 agent-sdk-custom-tools 这个 Skill 之后，它更容易走到一条更对路的回答方式：

  1. 先解释 custom tool 是什么
  2. 再解释 tool() 怎么定义
  3. 再解释 createSdkMcpServer() 怎么包装
  4. 再解释 mcpServers 怎么注册到 query()
  5. 最后再讲 Redis、数据库、HTTP API 这些真实使用场景

所以这里 Skill 带来的帮助不是“替你查 Redis”，而是：

让 Claude 更容易正确带你把“查 Redis 这个能力”接出来。

以 session 为例，Skill 又帮了什么

再举一个我觉得很典型的例子。

假设你问：

如果没有专项 Skill，Claude 很容易：

  • 只讲一两个参数
  • 讲不清 resume 和 continue 的差别
  • 忘了提 listSessions()、getSessionMessages() 这些配套能力

但有了 agent-sdk-sessions 这种 Skill 之后，它更容易直接进入会话这个主题本身：

  • resume 是什么
  • continue 是什么
  • session 怎么列出来
  • 历史消息怎么查
  • 什么场景适合继续，什么场景适合恢复指定会话

所以对于使用者来说，它的价值是：

你不需要自己反复把问题问得很精准，Skill 会帮 Claude 更容易进入正确专题。

比如你问：

这些问题本来就很容易互相串台。

这时候 agent-sdk-advanced 的作用就出来了。

它会让 Claude 更容易聚焦在这一层：

  • permissionMode
  • allowedTools
  • disallowedTools
  • canUseTool
  • hooks
  • outputFormat
  • 预算和轮次控制

对于使用者来说，这类 Skill 的帮助非常直接：

  • 少走弯路
  • 少漏关键参数
  • 少把不同主题搅在一起

真正的使用方式，不是“学会项目结构”，而是“学会提问题”

我觉得这是很多人第一次用 any-agent 时最容易忽略的点。

从使用者角度看，any-agent 不要求你先去背它的目录结构。

真正更重要的是：

你是否知道自己现在想问的是哪一类问题。

比如：

如果你想了解 Agent SDK 基础

就问：

  • “给我讲一下 Claude Agent SDK 的基本用法”
  • “怎么跑一个最小的 Agent SDK demo？”
  • “query() 返回的是什么？”

如果你想接工具

就问：

  • “怎么给 Claude Agent SDK 加 custom tools？”
  • “怎么把 Redis 查询封装成 tool？”
  • “createSdkMcpServer() 应该怎么写？”

如果你想做多轮会话

就问：

  • “怎么恢复 Claude Agent SDK 的 session？”
  • “resume 和 continue 有什么区别？”
  • “怎么查看历史会话？”

如果你想做高级配置

就问：

  • “怎么限制 Agent 的工具权限？”
  • “怎么做 structured output？”
  • “怎么控制最大轮次和预算？”

你会发现，Skill 对使用者最实际的帮助，其实就体现在这里：

同样是问问题，装了 Skill 之后，更容易一问就问到点上。

session-harvest 这种 Skill 对使用者又有什么意义

前面几个 Skill 都是“教你怎么做”的。

但 session-harvest 有点不一样，它更偏“把你已经摸索出的经验沉淀下来”。

它更适合这种时刻：

  • 你已经和 Claude 摸索出一套很顺的工作流
  • 你不想下次再从头讲
  • 你希望把这套经验变成未来可复用的 Skill

所以它对使用者的帮助不是：

  • “教你一个新知识点”

而是：

  • “帮你把一次成功经验固化下来”

这类 Skill 对长期重度使用者特别有价值。

我现在会怎么给别人解释 any-agent 的作用

如果现在让我只用一句使用者视角的话来解释，我会这么说：

any-agent 的作用，不是替你直接完成业务，而是让 Claude Code 在 Agent 相关主题下更容易走对方法。

再说得更直白一点：

  • 没有它，Claude 也许也能答
  • 有了它，Claude 更容易答得更准、更稳、更像一个真正懂这个专题的人

对使用者来说，最实用的价值是什么

最后我想把它再压缩成几个非常实际的点。

对使用者来说，装了 any-agent 之后，最实用的价值通常是：

1. 复杂主题更容易命中正确方向

尤其是 custom tools、session、advanced 这种很容易混的主题。

2. 提问成本更低

你不需要每次都把问题问得特别精确，Skill 会帮 Claude 更容易进入正确专题。

3. 回答更稳定

特别是在需要分步骤解释的时候，更不容易漏关键环节。

4. 更适合长期积累

像 session-harvest 这种 Skill，能把一次次临时对话里的经验逐步沉淀下来。

结语

我现在反而不太喜欢把 any-agent 介绍成一个“项目说明文”的对象。

因为对真正要用它的人来说，更重要的不是“它目录长什么样”，而是：

  • 装完之后我怎么问
  • 它到底帮我更好地解决了什么问题
  • 为什么我同样一句话，装了它之后更容易得到正确答案

如果从这个角度看，any-agent 的价值其实很清楚：

它不是替你做业务的系统，而是帮助 Claude 更稳定地成为某些专题下的好助手。
