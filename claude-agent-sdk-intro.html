Claude Agent SDK 入门：是什么、什么时候用、query、session 与 MCP / tools / agent

发表于 2026-04-05 分类于 技术分享 本文字数： 6.2k 阅读时长 ≈ 6 分钟

最近在系统整理 Claude Agent SDK。刚开始接触这套东西时，我最大的感受不是“代码很难”，而是“概念很容易缠在一起”。

很多文章会同时出现这些词：

  • query()
  • session
  • custom tools
  • MCP server
  • agent
  • skill

第一次读的时候，如果没有顺序，很容易一篇看完只记住了名词，没有真正形成心智模型。

所以这篇文章我不讲 Skill，不讲插件市场，也不讲 any-agent。我们只把 Claude Agent SDK 本身这一层拉直，用 5 个块来建立基础认知：

  1. 什么是 Claude Agent SDK
  2. 什么时候应该用它
  3. 最小 query() 调用怎么跑
  4. session 是什么，以及怎么做一个最小 demo
  5. 如何扩展出 MCP server、custom tools 和 agent

1. 什么是 Claude Agent SDK

如果一句话概括：

Claude Agent SDK 是一个让模型不只是“回答一句话”，而是能够带着上下文、工具和角色去执行任务的运行框架。

普通模型调用通常更像：

而 Claude Agent SDK 更像：

所以它和普通模型 API 最大的区别，不只是接口名字不一样，而是它多了一层“Agent 运行时”。

这个运行时通常包含：

  • 上下文
  • 会话能力
  • 工具能力
  • 角色能力
  • 权限控制
  • 流式消息

你可以把它简单理解成：

  • 普通模型 API 更像“问答接口”
  • Agent SDK 更像“任务执行框架”

2. 什么时候应该用它

并不是所有场景都需要 Claude Agent SDK。

如果你的需求只是：

  • 翻译
  • 润色
  • 单轮问答
  • 一次性总结

那普通模型 API 往往已经足够。

Claude Agent SDK 更适合这些场景：

  • 你希望模型能连续记住上下文
  • 你希望模型能调用你的工具
  • 你希望模型不是只聊天，而是真正“做事”
  • 你希望模型能分角色工作
  • 你希望对权限和行为边界做控制

换句话说，只要你的需求开始出现下面这些表达，就可以开始考虑 Agent SDK：

  • “帮我继续上一次分析”
  • “帮我查一下系统状态”
  • “帮我调用我自己的服务”
  • “帮我根据真实数据给出结论”
  • “帮我以研究员/审查员/运维助手的视角来处理”

比如这些就很典型：

  • 让 Agent 查询 Redis 里的任务状态，并给出中文总结
  • 让 Agent 查数据库中的业务记录，再给出排查建议
  • 让 Agent 继续上一次的研究上下文，而不是每次重新开始
  • 让 Agent 以不同角色输出不同风格的结论

这里最重要的理解是：

  • query() 启动的是一次 Agent 任务
  • 返回值不是一个普通 Promise，而是一个消息流
  • 你会陆续收到初始化信息、assistant 消息、最终结果

也就是说，它不是“我调一下，直接拿字符串”，而是“我启动一个过程，然后不断接收过程中的消息”。

options 里最先该看懂的几个参数

第一次写 demo 时，其实不需要把所有参数都记全。最先看懂下面几个就够了。

cwd

告诉 Agent 当前任务运行在什么目录上下文里。

permissionMode

决定 Agent 的权限模式。

入门 demo 里这个模式很适合，因为它更保守，更偏“先规划，再执行”。

maxTurns

限制这次任务最多跑几轮。

model

如果你有自己的模型路由，也可以显式指定。

第一个 demo 应该重点观察什么

我建议第一次跑 query() 时，至少确认这几件事：

  1. 有没有正常启动
  2. 有没有输出 assistant 消息
  3. 有没有最终 result
  4. 当前模型是不是你预期的那个

只要这几件事打通，你就已经真正把 Claude Agent SDK 跑起来了。

为什么它值得入门阶段就掌握

很多真实任务天生就是多轮的：

  • 研究一个项目
  • 排查一个问题
  • 写一个方案
  • 跟着一个分析过程不断补充条件

如果每次都从零开始，模型虽然也能回答，但体验会非常碎。

session 解决的就是这个问题。

最常见的几类动作包括：

  • continue

  • resume
    恢复指定 session
  • listSessions()
    列出历史 session
  • getSessionMessages()
    查看某个 session 的消息记录

5. 如何扩展出 MCP server、custom tools 和 agent

当你已经理解：

  • query() 是任务入口
  • session 是连续上下文能力

下一步最值得进入的，就是工具和角色这一层。

5.1 custom tools 是什么

最直接的理解就是：

把你自己的能力暴露给 Agent 调用。

这个能力可以是：

  • 查数据库
  • 查 Redis
  • 调 HTTP API
  • 调内部服务
  • 查日志
  • 改任务状态

所以 custom tool 本质上不等于“数据库工具”。数据库只是其中一种使用场景。

但如果把内部逻辑换成：

  • 查 MySQL
  • 查 PostgreSQL
  • 查 Redis

那它就能变成真实业务能力。

5.2 MCP server 是什么

MCP 是 Model Context Protocol。

可以把它理解成：

给模型提供工具的一套标准接口。

而 MCP server 就是按这套协议，把一组工具暴露给模型调用的服务层。

所以：

  • 它不只是浏览器、Postman 这种大工具才会有
  • 你自己定义的一组业务 tool，也完全可以组成一个 MCP server

比如：

  • Redis 任务状态查询 MCP server
  • 订单查询 MCP server
  • 日志检索 MCP server

5.4 agent 是什么

agent 更像“角色配置”。

比如你可以定义：

  • researcher
  • reviewer
  • ops-assistant

不同角色可以有不同的：

  • prompt
  • 工具
  • 模型
  • 最大轮次

一张表记住这一层关系

概念 它是什么 它解决什么问题
query() Agent 任务入口 启动一次任务并接收消息流
session 连续上下文能力 让 Agent 可以继续同一个任务，而不是每轮从零开始
custom tool 具体能力 让 Agent 能查数据、调系统、改状态
MCP server 工具组织方式 把一组能力按统一协议暴露给 Agent
agent 角色配置 让不同任务有不同风格和边界
