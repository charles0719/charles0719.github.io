如何从零创建一个简单 Skill：目录结构、SKILL.md 和最小可用模板

发表于 2026-04-06 更新于 2026-04-05 分类于 技术分享 本文字数： 4k 阅读时长 ≈ 4 分钟

最近在研究 Claude Code 里的 Skill 机制，发现很多资料都讲到了 Skill 的作用，但第一次真正准备自己动手写时，脑子里最常见的问题其实是这些：

  • 一个 Skill 最少要包含哪些东西？
  • SKILL.md 里到底该写什么？
  • description 为什么这么重要？
  • 我看到有些地方还有 steps、references、scripts，这些是不是必须的？

这篇文章就不再泛泛讲概念，而是直接回答一个更实用的问题：

如果你现在就想创建一个最简单、但真的能工作的 Skill，应该怎么做。

先说结论：一个 Skill 最少只需要两样东西

如果你只追求“先跑起来一个最小可用版本”，那真正的必需项其实非常少：

description 为什么特别重要

这是我觉得第一次写 Skill 时最该重视的一点。

description 不是一个普通说明字段，它其实是 Skill 最关键的触发入口之一。

Claude 会读这段描述，来判断：

  • 什么时候应该激活这个 Skill
  • 这个 Skill 适合处理什么问题

  • 写清楚用户会怎么问
  • 写清楚这个 Skill 适合在哪些场景下触发

所以你可以把 description 理解成：

Skill 的主要触发说明。

正文该写什么

很多人第一次写 Skill 时，会把正文写成一段很长的背景介绍。

但我现在更推荐一种更实用的写法：

  • 少讲大道理
  • 多写“触发后该怎么做”

也就是说，正文最好偏流程，而不是散文。

例如：

这里你看到的这些 step，并不是什么必须的 schema 字段，它们只是正文里的一种好写法。

它的意义是：

  • 帮 Claude 明白“触发后先做什么，再做什么”
  • 让 Skill 更像一份工作流程说明，而不是一段模糊说明

所以如果你在别人的 Skill 里看到：

  • Step 1
  • Step 2
  • Step 3

不用把它想成“必须配置项”，它更像是一种非常推荐的正文写法。

一个 Skill 可以再包含哪些东西

当你的 Skill 不再只是最小版本时，通常还会加入一些可选目录。

比较常见的是：

它们分别解决不同问题。

scripts/

适合放可执行脚本。

什么时候该加：

  • 这件事总是反复做
  • 你不想每次都让 Claude 重新生成同样的脚本
  • 你希望结果更稳定

例如：

  • 查询某个 API 的脚本
  • 批量处理文件的脚本
  • 固定格式转换脚本

references/

适合放参考资料。

什么时候该加：

  • 有一些内容很重要，但不适合全塞进 SKILL.md
  • 你希望 Claude 按需读取，而不是每次都把全文塞进上下文

例如：

  • 数据库 schema
  • API 文档
  • 公司内部规则
  • 某个业务模块的详细说明

assets/

适合放资源文件。

例如：

  • 模板
  • 图标
  • 示例文件
  • 前端样板代码

一个很推荐的创建顺序

如果你现在准备从零创建一个 Skill，我会建议按下面这个顺序来。

第一步：先想清楚用户会怎么问

这是最重要的一步。

不要先写目录，不要先写代码，先想：

  • 用户会怎么描述这个需求？
  • 哪些提问应该命中这个 Skill？
  • 命中之后，Claude 应该先做什么？

如果这一步不清楚，后面 description 就很容易写得很空。

第二步：先写最小版 SKILL.md

先不要急着上 scripts/、references/、assets/。

先把最小版写出来：

  • name
  • description
  • 一段清晰的正文流程

只要这个最小版能讲清楚触发条件和处理流程，就已经比很多“又大又空”的 Skill 强了。

第三步：再决定要不要拆资源

当你发现正文越来越长时，再考虑拆出去：

  • 可执行逻辑放 scripts/
  • 参考资料放 references/
  • 模板资源放 assets/

这一步的关键不是“目录越完整越专业”，而是：

让 SKILL.md 保持精炼。

写 Skill 时，我觉得最容易踩的坑

1. description 写得太空

这是第一大坑。

如果 description 不具体，Skill 触发效果通常就不会太好。

2. 正文全是背景，没有流程

Skill 正文不是写“这个主题有多重要”，而是写“被触发后 Claude 应该怎么做”。

3. 什么都塞进 SKILL.md

如果 schema、API 文档、示例、脚本全塞进去，正文会越来越胖，后面也不容易维护。

4. 一上来就想做很大

我会非常建议第一次先做最小版。

真的能触发、能工作，再慢慢加 references/、scripts/。

用一句话总结 Skill 的创建逻辑

如果让我把整件事压成一句话，我会这样说：

先用 name + description 定义“什么时候触发”，再用 SKILL.md 正文定义“触发后怎么做”，复杂资料再拆到 references/、scripts/、assets/。

结语

第一次创建 Skill，其实不用把它想得太重。

你不需要一开始就做一个很完整的大仓库，也不需要一开始就准备特别复杂的目录。

更实用的做法反而是：

  • 先想清楚它要解决什么问题
  • 先写一个最小可用的 SKILL.md
  • 先把触发和流程写准
  • 再根据真实使用慢慢迭代

我觉得这才是最适合真正上手的节奏。
